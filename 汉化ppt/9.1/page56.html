Refactoring at Google
9.1 Code Smells  Refactoring
▪ "Common reasons not to do it are incorrect:
– a) 'Don't have time; features more important' -- You will pay more cost, time
in adding features (because it's painful in current design), fixing bugs
(because bad code is easy to add bugs into), ramping up others on code
base (because bad code is hard to read), and adding tests (because bad
code is hard to test), etc.
– b) 'We might break something' -- Sign of a poor design from the beginning,
where you didn't have good tests. For same reasons as above, you should
fix your testing situation and code.
– c) 'I want to get promoted and companies don't recognize refactoring work' --
This is a common problem. Solution varies depending on company. Seek
buy-in from your team, gather data about regressions and flaws in the
design, and encourage them to buy-in to code quality."
▪ "An important line of defense against introducing new bugs in a
refactor is having solid unit tests (before the refactor)."
-- Victoria Kirst, Google
